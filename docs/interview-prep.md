# 면접 대비 - 기술 질문 & 답변

## Redis 대기열

**Q. 왜 DB 대신 Redis로 대기열을 구현했나요?**

DB로 구현하면 대기열 조회 시 매번 `COUNT` 쿼리가 발생하고, 대규모 트래픽에서 DB 병목이 생깁니다.
Redis는 인메모리라 응답 속도가 빠르고, ZSET 자료구조가 score 기반 정렬을 기본 제공해서 선착순 대기열에 적합합니다.
또한 TTL을 활용해 토큰 만료를 별도 스케줄러 없이 처리할 수 있어 선택했습니다.

---

**Q. ZSET과 String을 왜 분리해서 사용했나요?**

역할이 다릅니다.
- **ZSET (`waiting-token`)** — 대기 순서를 score(타임스탬프)로 관리. 선착순 보장.
- **String (`active-token:{token}`)** — 활성 토큰 저장, TTL 10분 설정으로 자동 만료.

ZSET은 TTL 설정이 불가능하고, String은 정렬이 불가능합니다. 각자의 특성에 맞게 역할을 분리했습니다.

---

**Q. Lua 스크립트를 왜 사용했나요?**

토큰 활성화는 "ZSET에서 꺼내기 → String에 저장" 두 단계로 이루어지는데,
이 사이에 다른 요청이 끼어들면 중복 활성화가 발생할 수 있습니다.
Redis는 단일 스레드이지만 여러 명령어를 보낼 경우 중간에 다른 클라이언트 요청이 처리될 수 있습니다.
Lua 스크립트는 Redis에서 원자적으로 실행되어 이 문제를 해결하고, 네트워크 RTT도 1회로 줄입니다.

## 동시성 제어

**Q. 낙관적 락과 비관적 락 중 왜 낙관적 락을 선택했나요?**

좌석 예약은 **충돌이 드물게 발생**하는 시나리오입니다. (같은 좌석을 동시에 예약하는 경우)
비관적 락은 충돌 여부와 관계없이 항상 DB에 락을 걸어 대기가 발생하고 성능이 떨어집니다.
낙관적 락은 충돌이 없으면 락 오버헤드가 전혀 없고, 충돌 시에만 `OptimisticLockException`으로 재처리합니다.
트래픽이 많아도 정상 케이스에서 성능 저하가 없다는 장점으로 선택했습니다.

---

**Q. 잔액 차감에는 왜 분산 락을 사용했나요?**

잔액 차감은 "조회 → 계산 → 저장" 흐름이라 동시 요청 시 같은 잔액을 두 번 읽어 중복 차감이 발생할 수 있습니다.
낙관적 락으로도 해결 가능하지만, 잔액 같은 경우 충돌 시 재시도 로직이 복잡해지고 사용자 경험도 나빠집니다.
분산 락은 유저 단위로 직렬화하여 확실하게 중복 요청을 차단합니다.
또한 Redisson의 `waitTime=0` 설정으로 중복 요청 자체를 즉시 거부합니다.

---

**Q. 분산 락 leaseTime을 3초로 설정한 이유는?**

leaseTime은 락을 자동 해제하는 시간입니다.
너무 짧으면 비즈니스 로직 수행 중 락이 해제되어 다른 요청이 진입할 수 있고,
너무 길면 장애 시 다른 요청이 오래 대기합니다.
잔액 차감 로직이 DB 쓰기 1회로 간단하기 때문에 3초면 충분하다고 판단했습니다.

## Kafka & Saga 패턴

**Q. 왜 Kafka를 사용했나요?**

결제 플로우는 잔액 차감 → 예약 확정 → 결제 생성 순서로 여러 도메인이 연관됩니다.
단일 트랜잭션으로 묶으면 서비스가 커질수록 트랜잭션 범위가 넓어지고 DB 락 경합이 심해집니다.
Kafka를 사용하면 각 도메인이 독립적으로 처리되고, 실패 시 보상 트랜잭션으로 롤백할 수 있어 선택했습니다.

---

**Q. Saga 패턴이 뭔가요? 왜 코레오그래피 방식을 선택했나요?**

Saga 패턴은 분산 트랜잭션을 여러 개의 로컬 트랜잭션으로 나누고, 실패 시 보상 트랜잭션으로 이전 상태로 복구하는 패턴입니다.

두 가지 방식이 있습니다:
- **오케스트레이션** — 중앙 오케스트레이터가 각 서비스를 순서대로 호출
- **코레오그래피** — 각 서비스가 이벤트를 발행하고, 다른 서비스가 구독해서 처리

코레오그래피를 선택한 이유는 중앙 오케스트레이터 없이 서비스 간 결합도를 낮출 수 있기 때문입니다.
각 서비스가 자신의 토픽만 구독하면 되므로 확장에도 유리합니다.

---

**Q. 보상 트랜잭션은 어떻게 구현했나요?**

결제 실패 시 Kafka `balance-rollback` 토픽에 이벤트를 발행하면,
`BalanceConsumer`가 구독해서 차감했던 잔액을 다시 증가시키는 방식으로 구현했습니다.

```
결제 실패 → PaymentConsumer → reservation-rollback 발행
                             → balance-rollback 발행
ReservationConsumer → 예약 상태 복원
BalanceConsumer     → 잔액 환불
```

## @TransactionalEventListener

**Q. 왜 AFTER_COMMIT을 사용했나요?**

처음에는 트랜잭션 안에서 `queueTokenService.expireToken()`을 직접 호출했습니다.
그런데 Redis 명령어는 Spring의 `@Transactional` 범위 밖이라 DB가 롤백되어도 Redis 토큰은 이미 삭제된 상태가 됩니다.
그러면 사용자가 재시도해도 토큰이 없어서 요청 자체가 불가능해집니다.

`AFTER_COMMIT`을 사용하면 DB 커밋이 성공한 이후에만 Redis 삭제가 실행되고,
DB 롤백 시에는 이벤트 자체가 발행되지 않아 토큰이 보존됩니다.

---

**Q. @Transactional 안에 있는데 왜 Redis는 롤백이 안 되나요?**

Spring의 `@Transactional`은 `JpaTransactionManager`가 JDBC 커넥션의 commit/rollback만 제어합니다.
Redis는 완전히 별도의 서버이고, `redisTemplate.delete()`는 호출 즉시 Redis 서버에 명령이 전송됩니다.
트랜잭션 매니저가 Redis를 추적하지 않기 때문에 롤백 메커니즘이 없습니다.

## 인덱스 & 성능

**Q. 어떤 기준으로 인덱스를 설계했나요?**

- 조회 조건에 자주 쓰이는 컬럼 (예: `user_id`, `concert_schedule_id`)
- 카디널리티가 높은 컬럼을 복합 인덱스 선두에 배치
- `SELECT *` 대신 필요한 컬럼만 조회하여 커버링 인덱스 활용 가능하도록

인덱스를 무조건 많이 걸면 쓰기 성능이 저하되므로, 실제 쿼리 패턴을 보고 최소한으로 적용했습니다.

## 아키텍처

**Q. DDD 레이어드 아키텍처를 선택한 이유는?**

도메인 로직이 특정 프레임워크나 인프라에 의존하지 않도록 하기 위해서입니다.
`domain` 레이어는 순수 Java 객체로만 구성되고, JPA나 Redis 같은 구현체는 `infrastructure`에 격리됩니다.
덕분에 테스트 시 Mock 처리가 쉽고, 나중에 JPA를 다른 기술로 교체해도 도메인 로직에 영향이 없습니다.

---

**Q. Facade 패턴을 사용한 이유는?**

예약, 결제 같은 플로우는 여러 도메인 서비스를 조합해야 합니다.
각 컨트롤러에서 직접 여러 서비스를 호출하면 로직이 분산되고 트랜잭션 관리가 어렵습니다.
Facade가 오케스트레이션 역할을 맡아 `@Transactional`을 단일 지점에서 관리하고, 컨트롤러는 단순 위임만 합니다.
