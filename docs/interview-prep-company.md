## 맵인어스 — 운행기록분석시스템

### 배차 예약 등록 기능

**Q. 배차 예약 등록 기능을 왜 만들었나요?**

대용량 배차 데이터를 등록할 때 처리 시간이 길어 사용자가 실시간으로 대기해야 하는 UX 문제가 있었습니다.
Spring Scheduler를 활용해 등록 작업을 예약 실행하도록 바꿔서 사용자가 대기하지 않아도 되도록 개선했습니다.
오류 발생 시에는 오류 가이드를 자동으로 제공해 운영팀이 직접 대응할 수 있도록 했습니다.

> **꼬리 질문 대비**
>
> - **예약 작업이 실패하면 어떻게 처리했나요?**
>   오류 발생 시 오류 코드와 원인을 DB에 저장하고 운영자 화면에 가이드 메시지로 표시했습니다.
>   자동 재시도보다 운영자가 확인 후 수동 재실행하는 방식을 택했습니다. 배차 데이터 특성상 자동 재시도 시 중복 등록 리스크가 있었기 때문입니다.
>
> - **@Async 사용 시 주의할 점은?**
>   `@Async` 메서드는 별도 스레드에서 실행되기 때문에 `@Transactional`이 전파되지 않습니다.
>   호출자의 트랜잭션과 분리되므로, 비동기 작업 내에서 별도 트랜잭션을 열어야 합니다.
>   또한 기본 스레드풀(`SimpleAsyncTaskExecutor`)은 스레드를 무한 생성해 OOM 위험이 있어
>   `ThreadPoolTaskExecutor`로 풀 크기를 제한해야 합니다.
>
> - **Spring Batch는 왜 안 썼나요?**
>   Spring Batch는 대용량 데이터를 청크 단위로 반복 처리하거나 Step 간 의존성이 복잡한 경우에 적합합니다.
>   이 기능은 단순히 "등록 요청을 비동기로 실행"하는 수준이라 Scheduler + Async 조합이 충분했습니다.
>   불필요한 복잡도를 추가하지 않으려 했습니다.

**Q. Spring Scheduler를 선택한 이유는?**

배차 등록은 주기적으로 반복 실행하는 구조가 아니라, 사용자가 요청한 시점에 비동기로 처리하는 구조입니다.
별도의 배치 프레임워크(Spring Batch)를 도입하기엔 오버엔지니어링이고,
`@Scheduled`와 `@Async`를 조합해 간단하게 비동기 예약 실행을 구현했습니다.

> **꼬리 질문 대비**
>
> - **@Scheduled와 Quartz의 차이는?**
>   `@Scheduled`는 단일 인스턴스 환경에서 간단한 주기 실행에 적합합니다.
>   Quartz는 클러스터 환경에서 여러 인스턴스 간 중복 실행을 방지하는 분산 스케줄링을 지원합니다.
>   서버가 한 대인 환경이었기 때문에 `@Scheduled`로 충분했습니다.
>
> - **멀티 서버 환경이었다면 어떻게 했을 것 같나요?**
>   Quartz + DB 기반 잡 락 or Redis 분산 락으로 한 서버에서만 실행되도록 보장했을 것입니다.

### EON Mail 시스템

**Q. EON Mail을 직접 설계했나요?**

네, 요구사항 정의부터 설계, 개발까지 주도했습니다.
공단 SMTP 서버와 연동해 공식 메일링 채널을 구성하고,
관리자 페이지에서 발신 주소·주기 설정, 발송 이력 조회, 수신 확인, 재발송까지 전체 관리 기능을 개발했습니다.

> **꼬리 질문 대비**
>
> - **SMTP 직접 연동 vs SendGrid 같은 외부 서비스의 차이는?**
>   외부 메일 서비스(SendGrid, AWS SES)는 발송 실패 처리, 수신 거부 관리, 통계 등을 제공하지만 비용이 발생합니다.
>   공단 내부 SMTP 서버를 사용한 건 보안 정책과 공식 발신 주소 요건 때문이었습니다.
>   내부 서버라 발송 실패 시 재시도 로직과 이력 관리를 직접 구현해야 했습니다.
>
> - **수신 확인은 어떻게 구현했나요?**
>   메일 본문에 1x1 투명 이미지(트래킹 픽셀)를 삽입하고, 이미지 요청이 들어오면 수신 확인으로 처리하는 방식을 사용했습니다.
>   단, 이미지 자동 차단 클라이언트에서는 수신 확인이 안 될 수 있어 참고 지표로만 활용했습니다.

**Q. 대용량 메일링 확장은 어떻게 구현했나요?**

운행기록 분석 결과를 정기적으로 대규모 수신자에게 발송해야 했습니다.
업무 프로세스를 분석해 최적 발송 주기를 결정하고,
JAR 빌드 후 크론 작업(cron job)으로 정기 발송을 자동화했습니다.
웹 애플리케이션 서버와 분리된 독립 프로세스로 운영해 서버 부하를 줄였습니다.

> **꼬리 질문 대비**
>
> - **왜 웹 서버 안에 두지 않고 별도 JAR로 분리했나요?**
>   대용량 메일 발송은 수백~수천 건을 처리하는 작업이라 웹 서버의 스레드풀을 점유할 위험이 있었습니다.
>   별도 프로세스로 분리하면 웹 서버 응답성에 영향을 주지 않고, 발송 스케줄도 독립적으로 관리할 수 있습니다.
>
> - **발송 중 서버가 죽으면 어떻게 되나요?**
>   발송 이력을 DB에 상태로 관리해서 미발송 건을 식별할 수 있게 했습니다.
>   크론 재실행 시 미완료 건부터 재처리할 수 있는 구조로 설계했습니다.

### 대용량 배차 등록 성능 개선

**Q. 성능 병목 원인을 어떻게 찾았나요?**

배차 파일 등록 시간이 2분 이상 걸리는 문제가 있었습니다.
구간별로 시간을 측정해 병목 지점을 세 군데 확인했습니다.
1. 엑셀 파싱 단계 — DOM 방식이 대용량 파일 전체를 메모리에 올려 OOM 위험
2. 검증 쿼리 단계 — 인덱스 없이 Full Table Scan 발생
3. DB 저장 단계 — 건별 Commit으로 트랜잭션 오버헤드 과다

> **꼬리 질문 대비**
>
> - **구간별 측정을 어떻게 했나요?**
>   `System.currentTimeMillis()` 또는 `StopWatch`로 각 단계 시작/종료 시간을 로그로 출력했습니다.
>   APM 도구 없이도 단계별 소요 시간을 빠르게 파악할 수 있었습니다.
>
> - **세 가지 중 어느 게 가장 큰 병목이었나요?**
>   인덱스 개선이 가장 효과가 컸습니다. 검증 쿼리가 건수만큼 반복 실행되는 구조라
>   Full Table Scan이 누적되면서 전체 시간의 대부분을 차지했습니다.

**Q. DOM → SAX 변경으로 얼마나 개선됐나요?**

DOM은 파일 전체를 메모리에 로드하지만, SAX는 이벤트 기반으로 스트리밍 파싱합니다.
대용량 엑셀 파일에서 메모리 사용량이 대폭 줄었고, OOM 리스크도 제거됐습니다.
단독 개선 수치보다 세 가지를 함께 적용해 최종적으로 2분 → 30초, 약 75% 단축했습니다.

> **꼬리 질문 대비**
>
> - **SAX의 단점은?**
>   SAX는 순차 스트리밍 방식이라 특정 셀을 임의 접근(random access)할 수 없습니다.
>   이전에 읽은 데이터를 다시 참조하려면 직접 상태를 관리해야 해서 코드가 복잡해집니다.
>   DOM처럼 트리 구조를 바로 활용하기 어렵습니다.
>
> - **StAX는 알고 있나요?**
>   SAX는 이벤트를 콜백으로 받는 push 방식이고, StAX는 필요할 때 직접 다음 이벤트를 당겨오는 pull 방식입니다.
>   StAX가 코드 흐름 제어가 더 직관적이지만 엑셀 파싱은 Apache POI의 SAX 방식(`XSSF + ContentHandler`)이
>   이미 잘 지원되어 SAX를 사용했습니다.
>
> - **실제로 OOM이 발생한 적 있었나요?**
>   운영 중 특정 크기 이상의 파일을 올릴 때 서버가 멈추는 현상이 있었고, 그게 OOM이 원인이었습니다.
>   SAX 전환 후 같은 파일을 올려도 정상 처리됐습니다.

**Q. 인덱스는 어떤 기준으로 설계했나요?**

배차 검증 쿼리의 WHERE 조건 컬럼을 분석해 적합한 인덱스를 추가했습니다.
카디널리티가 높은 컬럼을 복합 인덱스 선두에 배치하고,
쓰기 성능 저하를 최소화하기 위해 실제 쿼리 패턴에 맞는 최소한의 인덱스만 적용했습니다.

> **꼬리 질문 대비**
>
> - **인덱스가 많으면 어떤 문제가 생기나요?**
>   INSERT/UPDATE/DELETE 시 인덱스도 함께 갱신해야 해서 쓰기 성능이 저하됩니다.
>   또한 인덱스 자체가 디스크 공간을 차지합니다.
>   쿼리 옵티마이저가 잘못된 인덱스를 선택할 가능성도 있어 무조건 많이 걸면 역효과가 납니다.
>
> - **카디널리티가 낮은 컬럼에 인덱스를 걸면?**
>   예를 들어 `성별(M/F)`처럼 값의 종류가 2개뿐인 컬럼은 인덱스로 절반밖에 걸러내지 못합니다.
>   Full Table Scan보다 나을 게 없어서 옵티마이저가 인덱스를 무시하는 경우도 많습니다.
>
> - **커버링 인덱스란?**
>   쿼리에서 필요한 모든 컬럼이 인덱스에 포함되어 있어, 테이블 본문을 조회하지 않고 인덱스만으로 결과를 반환하는 방식입니다.
>   디스크 I/O가 줄어 조회 성능이 크게 향상됩니다.

**Q. 중복 배차 검사 로직을 왜 재설계했나요?**

기존 로직은 파일 내 중복과 DB에 이미 등록된 중복을 별도로 처리하지 않아 정합성 문제가 있었습니다.
검사 로직을 통합해 파일 내 중복과 기등록 중복을 한 번에 처리하도록 재설계했습니다.
이 과정에서 불필요한 쿼리 호출도 줄어 성능도 함께 개선됐습니다.

> **꼬리 질문 대비**
>
> - **파일 내 중복은 어떻게 검출했나요?**
>   파싱하면서 고유 키(배차 식별자)를 `HashSet`에 누적해, 이미 존재하면 중복으로 처리했습니다.
>   O(1) 조회라 건수가 많아도 성능에 영향이 없었습니다.
>
> - **기등록 중복은 건별로 쿼리를 날렸나요?**
>   처음엔 건별로 SELECT 쿼리를 날렸는데 이게 병목이었습니다.
>   파일에서 읽은 키를 리스트로 모아 `IN` 쿼리 한 번으로 기등록 여부를 일괄 조회하도록 변경했습니다.
>   쿼리 호출 횟수를 N번 → 1번으로 줄였습니다.

### 도로기반 데이터 신청 기능

**Q. 이 기능을 왜 개발했나요?**

기존 시스템은 운행지역(행정구역) 단위로만 데이터를 신청할 수 있어
특정 도로 구간의 데이터만 필요한 경우에도 지역 전체 데이터를 받아야 했습니다.
지도에서 직접 도로를 선택해 필요한 구간만 신청할 수 있는 기능을 새로 설계·개발했습니다.

> **꼬리 질문 대비**
>
> - **기존 지역 단위 방식의 구체적인 문제는?**
>   예를 들어 서울 특정 도로 1km 구간 데이터만 필요한데, 서울 전체 운행 데이터를 받아야 했습니다.
>   데이터 용량이 불필요하게 크고, 담당자가 직접 필터링해야 하는 후처리 부담이 있었습니다.

**Q. PostGIS pgRouting과 Dijkstra를 어떻게 활용했나요?**

시점과 종점을 지도에서 선택하면, PostGIS의 pgRouting 확장 기능으로
Dijkstra 알고리즘을 실행해 최단거리 경로를 계산합니다.
cost 기준은 `거리 / 링크 최대속도`로 설정해 단순 거리가 아닌 주행 시간 기반 최단경로를 구했습니다.
데이터는 국가교통정보센터의 표준노드링크를 기반으로 사용했습니다.

> **꼬리 질문 대비**
>
> - **Dijkstra 알고리즘이 뭔가요?**
>   하나의 출발점에서 다른 모든 노드까지의 최단경로를 구하는 알고리즘입니다.
>   우선순위 큐(min-heap)로 현재까지 비용이 가장 낮은 노드를 먼저 탐색합니다.
>   시간복잡도는 O((V + E) log V)이며, 음수 가중치가 없는 그래프에서 동작합니다.
>
> - **Dijkstra vs A* 차이는?**
>   Dijkstra는 출발점에서 모든 방향으로 균등하게 탐색합니다.
>   A*는 목적지 방향으로의 휴리스틱(예: 직선거리)을 추가해 불필요한 탐색을 줄입니다.
>   도로 네트워크처럼 목적지가 명확한 경우 A*가 더 빠르지만,
>   pgRouting이 Dijkstra를 기본 제공해서 그대로 활용했습니다.
>
> - **음수 가중치가 있으면 Dijkstra를 쓸 수 없나요?**
>   맞습니다. 음수 가중치가 있으면 Dijkstra는 잘못된 결과를 낼 수 있습니다.
>   이 경우 Bellman-Ford 알고리즘을 사용해야 합니다.
>   도로 네트워크는 거리/속도 기반이라 항상 양수여서 Dijkstra가 적합했습니다.
>
> - **표준노드링크가 뭔가요?**
>   국가교통정보센터(ITS Korea)에서 제공하는 전국 도로망 공간 데이터입니다.
>   도로를 노드(교차점)와 링크(구간)로 구성한 그래프 형태로 제공되어
>   pgRouting의 Dijkstra 입력 데이터로 바로 활용할 수 있었습니다.

**Q. 50m 격자 중첩은 왜 했나요?**

선택된 도로 경로와 운행기록 데이터를 매칭할 때, 도로 전체 범위로 조회하면 데이터가 너무 많아집니다.
선택된 도로 위에 50m 격자를 중첩해 실제로 해당 도로를 지나는 격자만 추출하여
불필요한 데이터를 제거하고 추출 데이터 양을 최소화했습니다.

> **꼬리 질문 대비**
>
> - **격자 크기 50m는 어떻게 결정했나요?**
>   운행기록 데이터가 GPS 기반 50m 격자 단위로 집계된 시스템이었기 때문입니다.
>   기존 데이터 구조에 맞춰 동일한 격자 단위를 사용했습니다.
>
> - **PostGIS 공간 쿼리 성능은 어떻게 최적화했나요?**
>   공간 인덱스(`GIST 인덱스`)가 핵심입니다. 일반 B-Tree 인덱스는 공간 데이터를 처리할 수 없고,
>   GIST 인덱스는 공간 객체의 경계 박스(bounding box)로 1차 필터링해 연산 대상을 줄입니다.
>   `ST_Intersects` 같은 공간 함수는 GIST 인덱스를 자동으로 활용합니다.

## 사이다경제 — 결제 시스템

**Q. 결제 시스템에서 어떤 부분을 담당했나요?**

무통장 입금, 환불, 쿠폰, 수강권, 관리자 결제 등 다양한 결제 유형을 신규 구현했습니다.
결제는 예외 케이스가 많아 로직 분기 처리와 테스트 케이스 작성에 공을 들였습니다.
이후 결제 로직 리팩토링과 응답 성능 개선도 함께 진행했습니다.

> **꼬리 질문 대비**
>
> - **결제에서 멱등성(idempotency)은 어떻게 보장했나요?**
>   결제 요청에 고유 주문 ID를 부여하고, 같은 주문 ID로 중복 요청이 들어오면
>   이미 처리된 결과를 그대로 반환했습니다.
>   네트워크 오류로 클라이언트가 재시도하더라도 중복 결제가 발생하지 않도록 했습니다.
>
> - **무통장 입금은 어떻게 처리했나요?**
>   주문 생성 시 가상계좌 또는 입금 대기 상태로 저장하고,
>   입금 확인(수동 또는 PG사 웹훅)이 완료되면 상태를 결제 완료로 업데이트하는 방식입니다.
>   입금 기한이 지나면 자동 만료 처리하는 스케줄러를 함께 구현했습니다.

---

**Q. 결제 로직에서 가장 신경 쓴 부분은?**

환불과 쿠폰 중복 적용 같은 예외 케이스입니다.
다양한 결제 유형이 혼합되는 경우(예: 쿠폰 + 수강권 동시 사용)의 분기를 명확히 처리하고,
각 케이스별 테스트 케이스를 작성해 회귀 버그를 방지했습니다.

> **꼬리 질문 대비**
>
> - **환불 시 트랜잭션은 어떻게 처리했나요?**
>   환불은 결제 취소와 DB 상태 변경이 원자적으로 이루어져야 합니다.
>   PG사에 취소 요청 → 성공 응답 받은 후 DB 상태를 환불 완료로 변경했습니다.
>   PG사 응답 전 DB 변경을 하면, PG사 요청 실패 시 데이터 불일치가 발생하기 때문입니다.
>
> - **쿠폰 중복 사용은 어떻게 막았나요?**
>   쿠폰 사용 시 DB에서 `사용 여부` 플래그를 `사용됨`으로 업데이트하고,
>   동시 요청 시 중복 사용을 막기 위해 쿠폰 단위 락을 적용했습니다.

>   `CountDownLatch`와 `ExecutorService`로 멀티 스레드 환경을 시뮬레이션해
>   낙관적 락 충돌, 분산 락 동작을 검증했습니다.
>   실제 운영보다 제약이 있지만, 설계 의사결정 과정 자체가 핵심 경험이었습니다.

## 공통 질문

**Q. 회사에서 가장 기술적으로 어려웠던 경험은?**

대용량 배차 등록 성능 개선이었습니다.
문제를 처음 보고했을 때 "원래 느린 거다"라는 반응이 있었는데,
직접 구간별 측정을 해서 DOM 파싱, 인덱스, 배치 커밋 세 가지 병목을 찾아냈습니다.
각각 개선한 결과 2분 → 30초로 75% 단축해 설득했고,
이후 비슷한 성능 이슈가 생기면 제게 먼저 물어보는 흐름이 만들어졌습니다.

> **꼬리 질문 대비**
>
> - **설득하는 과정이 어렵지 않았나요?**
>   "원래 느린 거다"라는 말을 수용하지 않고, 수치로 먼저 증명했습니다.
>   감으로 말하지 않고 구간별 로그 시간을 보여주면서 어디서 얼마나 걸리는지 명확히 제시했고,
>   개선 전후 비교 수치를 보고 나서는 자연스럽게 설득됐습니다.

---

**Q. 기술을 선택할 때 기준이 무엇인가요?**

문제를 해결하기에 충분한 가장 단순한 기술을 우선합니다.
배차 예약에 Spring Scheduler를 쓴 것처럼, Spring Batch까지 안 가도 되는 상황에서
오버엔지니어링 없이 해결하는 걸 중요하게 생각합니다.
단, 동시성이나 데이터 정합성처럼 나중에 고치기 어려운 문제는 처음부터 제대로 설계합니다.

> **꼬리 질문 대비**
>
> - **그 기준으로 실패한 경험도 있나요?**
>   처음 결제 로직을 단순하게 짰다가 결제 유형이 늘어나면서 분기가 복잡해진 경험이 있습니다.
>   초기 설계 시 확장 가능성을 조금 더 고려했어야 했는데, 그 이후로 결제·정합성 관련 로직은
>   처음부터 구조를 잡고 들어가는 습관이 생겼습니다.

**Q. 협업할 때 어떤 방식으로 일했나요?**

공단 측 담당자, 운영팀, 개발팀 사이에서 요구사항이 불명확하게 전달되는 경우가 많았습니다.
초기 회의에서 놓친 요구사항을 각 담당자와 1:1로 확인하며 정리하고,
양쪽의 목표를 일치시킨 뒤 개발에 들어가는 방식으로 리스크를 줄였습니다.

> **꼬리 질문 대비**
>
> - **요구사항이 개발 중간에 바뀐 경우는?**
>   도로기반 데이터 신청 기능 개발 중 "경유지도 추가해달라"는 요청이 중간에 들어왔습니다.
>   pgRouting이 다중 경유지를 지원하는 구조라 큰 변경 없이 대응할 수 있었습니다.
>   초기 설계 시 확장 가능성을 고려해둔 덕분이었습니다.
