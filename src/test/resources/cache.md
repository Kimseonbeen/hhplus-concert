## 1. 주요 캐시 전략

### 🔹 1.1 Look-Aside
- **개념 및 동작 방식**: 데이터 요청 시 먼저 캐시를 조회하고, 없으면 DB에서 조회 후 캐시에 저장
- **장점**: 필요할 때만 캐시에 적재하여 불필요한 데이터 캐싱 방지
- **단점**: 최초 요청 시 캐시 미스(Cache Miss) 발생 가능
- **활용 사례**: 자주 변경되는 데이터

### 🔹 1.2 Lazy Loading (Read-Through)
- **개념 및 동작 방식**: 필요할 때 데이터를 캐시에 적재
- **장점**: 불필요한 데이터 캐싱 방지
- **단점**: 초기 접근 시 응답 속도 저하 가능
- **활용 사례**: 자주 사용되지만 변경이 적은 데이터

### 🔹 1.3 Write-Through
- **개념 및 동작 방식**: 데이터를 캐시와 DB에 동시에 기록
- **장점**: 데이터 일관성 유지
- **단점**: 쓰기 성능 저하 가능
- **활용 사례**: 자주 변경되지 않는 데이터 캐싱

### 🔹 1.4 Write-Around
- **개념 및 동작 방식**: 데이터를 DB에 먼저 기록하고, 필요할 때만 캐시에 적재
- **장점**: 불필요한 데이터 캐싱 방지
- **단점**: 최초 접근 시 캐시 미스(Cache Miss) 발생 가능
- **활용 사례**: 읽기보다 쓰기가 많은 시스템

### 🔹 1.5 Write-Back
- **개념 및 동작 방식**: 데이터를 캐시에 우선 저장하고, 일정 주기 후에 DB에 반영
- **장점**: 쓰기 성능 향상
- **단점**: 장애 발생 시 데이터 유실 가능
- **활용 사례**: 쓰기 작업이 많고 성능이 중요한 서비스

## 2. 캐시 전략 활용 방안

| 캐시 전략 | 활용 상황 |
|-----------|----------------------|
| **Look-Aside** | 캐시 적중률을 높이고 싶은 경우 |
| **Lazy Loading (Read-Through)** | 자주 조회하지만 갱신 빈도가 낮은 데이터 |
| **Write-Through** | 데이터 일관성이 중요한 경우 |
| **Write-Around** | 쓰기 빈도가 높은 경우 |
| **Write-Back** | 쓰기 성능이 중요한 경우 |

## 3. 캐시 스탬피드(Cache Stampede) 현상

### 🔹 개념
- 동일한 데이터의 캐시가 만료될 때, 다수의 요청이 동시에 DB를 조회하여 부하가 급증하는 현상

### 🔹 원인 및 발생 조건
- 캐시 만료 시점에 다수의 요청이 동시에 들어오는 경우
- 캐시를 초기화하는 과정에서 발생하는 동시성 문제

### 🔹 해결 방법
- **Early Expiration**: 실제 만료 시간 보다 일찍 갱신하여 만료 전에 미리 새로운 데이터를 준비
- **Lua Script**: SETNX와 EXPIRE를 하나의 작업으로 실행 및 중간 단계에서 실패 가능성 제거
- **데이터 더블 캐싱(Double Caching)**: 이전 데이터를 유지한 채로 새로운 데이터를 준비하는 방식

## 5.Redis 기반 대기열 관리 시스템 개선안

---

### 📌 개요
기존의 대기열 관리 시스템은 데이터베이스를 이용하여 직접 순번을 계산하고 활성화하는 방식이었습니다.   
그러나 이러한 방식은 응답 속도 저하와 부하 증가를 초래할 수 있습니다.   
이를 해결하기 위해 **Redis의 Sorted Set** 자료형을 활용하여 보다 효율적인 대기열 관리를 구현합니다.

---

###  주요 개선 사항

### ✅ 1. 대기열 토큰 생성

#### 🔹 기존 방식
- 데이터베이스에서 순번을 계산하여 활성 또는 대기 상태의 토큰을 발급
- 활성 토큰 개수 조회 시 **O(n)** 의 성능 저하 발생
- 전체적인 성능과 응답 속도가 저하됨

#### 🔹 개선 방식
- **Redis의 Sorted Set**을 활용하여 자동으로 순번을 계산하고 토큰을 발급
- 활성 토큰 개수는 **O(n)**, 대기 토큰 개수는 **O(1)** 성능으로 개선
- **TTL(10분) 설정**으로 활성 토큰을 자동 삭제하여 관리 효율성 증가

---

### ✅ 2. 대기열 토큰 상태 조회

#### 🔹 기존 방식
- Queue 테이블에서 직접 활성 및 대기 토큰을 조회.
- 데이터베이스 조회로 인해 성능 저하 발생 가능.

#### 🔹 개선 방식
- 활성 토큰을 **String 자료형**으로 저장하여 먼저 조회 (**O(n)**).
- 대기 토큰은 **Sorted Set**을 사용하여 빠르게 조회 (**O(1)**).
- 활성 토큰이 **100개 이하**로 제한되므로 **O(100) 이하의 성능 유지 가능**.
- 존재하지 않는 토큰 조회 시 **예외 처리 강화**.

---

### ✅ 3. 토큰 상태 전환 스케줄러

#### 🔹 기존 방식
- 활성 -> 만료 상태 전환을 위한 **별도 스케줄러 필요**
- 대기 -> 활성 전환을 위해 **데이터베이스 조회 후 업데이트 진행**

#### 🔹 개선 방식
- 활성 -> 만료 전환은 **Redis TTL**을 활용하여 자동 관리
- 대기 -> 활성 전환을 위한 **스케줄러는 1분마다 실행**
- 활성 토큰 개수를 확인하여 **100개보다 적을 경우 대기 토큰을 활성 상태로 이동**

---

### ✅ 4. 결제 완료 시 토큰 삭제

#### 🔹 기존 방식
- 결제 완료 시 데이터베이스에서 활성 토큰을 삭제

#### 🔹 개선 방식
- 기존 로직을 유지하면서 **Redis에서 활성 토큰을 삭제**하여 처리 속도 향상

---

### ✅ 5. 핵심 컴포넌트

```java
public class QueueTokenService {
    private static final long MAX_ACTIVE_TOKEN_COUNT = 100;  // 최대 활성 토큰
    private final QueueTokenRepository queueTokenRepository;
}

public class QueueTokenRedisRepositoryImpl {
    private static final String WAITING_TOKEN_PREFIX = "waiting-token";
    private static final String ACTIVE_TOKEN_PREFIX = "active-token";
    private static final Duration TOKEN_TTL = Duration.ofMinutes(10);
}
```

### 🔹 데이터 구조

#### ✅ 활성 토큰
- **저장 방식**: Redis String
- **키 형식**: `active-token:{token}`
- **TTL**: 10분
- **최대 개수**: 100개

#### ✅ 대기 토큰
- **저장 방식**: Redis Sorted Set
- **키**: `waiting-token`
- **Score**: 시스템 시간(milliseconds)

---

### ✅ 6. N, M 결정 규칙 정리
🔹 N (토큰 활성화 주기) 결정 규칙
1. 사용자 행동 시간 고려
   예약 완료까지 평균 시간: 5분(300초)  
   N은 예약 시간의 1/10 이하가 적절  
   → 300초 / 10 = 30초  
2. 시스템 안정성 고려
   너무 짧은 주기 → 시스템 부하 증가  
   너무 긴 주기 → 사용자 경험 저하  
   → 30초가 적절한 균형점  
   ∴ N = 30초로 결정

🔹 MAX_ACTIVE_TOKEN_COUNT (동시 활성화 허용 토큰 수) 결정 규칙
1. 좌석 수 기반 계산  
   총 좌석 수: 50석  
   적정 경쟁률: 2.5~3배  
   → 50석 * 3 = 150개  
2. 시스템 처리 능력 고려  
   동시 접속자 처리 가능 여부  
   예약 프로세스 부하 감당 가능 여부  
   ∴ MAX_ACTIVE_TOKEN_COUNT = 150으로 결정  

🔹 설정 효과  

✅ 30초마다 체크 → 부족한 만큼만 토큰 활성화  
✅ 항상 150개 수준의 활성 토큰 유지 → 3:1 경쟁률 유지  
✅ 5분 동안 토큰 유효  
✅ 사용자는 평균 2.5분 정도 대기 예상  

🔹 운영 중 조정 가능성
- 예약 성공률이 너무 낮으면 → MAX_ACTIVE_TOKEN_COUNT 감소 
- 대기 시간이 길다는 피드백이 많으면 → ACTIVATION_INTERVAL 감소  
- 시스템 부하가 높으면 → ACTIVATION_INTERVAL 증가

### 🎯 기대 효과

- 데이터베이스 연산 부담을 줄이고 **Redis를 활용하여 성능 최적화**
- **빠른 읽기/쓰기 속도**로 대기열 처리 속도 개선
- **TTL을 활용한 자동 토큰 정리**로 관리 효율성 증가
- 활성 토큰 수를 **100개 이하**로 유지하여 **최적의 성능 보장**
