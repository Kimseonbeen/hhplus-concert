# MSA 전환에 따른 트랜잭션 처리 한계

## 1. 개요
MSA(Microservices Architecture) 환경에서 각 도메인은 독립적인 배포 단위와 데이터베이스를 가진다. 이러한 구조에서는 기존의 단일 트랜잭션 방식으로는 처리할 수 없는 한계가 존재한다. 본 문서에서는 MSA 환경에서 발생하는 트랜잭션 처리의 한계점을 분석하고, 이에 대한 해결 방안을 제시한다.

## 2. 기존 모놀리식 아키텍처의 트랜잭션 처리 방식
모놀리식 아키텍처에서는 일반적으로 **단일 데이터베이스를 사용**하며, 하나의 트랜잭션 안에서 여러 서비스 로직이 수행될 수 있다. 다음과 같은 특징을 가진다:

- **ACID(Atomicity, Consistency, Isolation, Durability) 보장**
- **단일 트랜잭션 내에서 여러 서비스의 데이터 일관성 유지 가능**
- **하나의 서비스에서 장애가 발생하면 롤백(rollback)이 용이**

### 예제 코드: 기존 모놀리식 트랜잭션
```java
@Transactional
public void 예약결제() {
    잔액서비스.잔액감소();
    예약서비스.예약완료();
    결제서비스.결제처리();
    토큰서비스.만료처리();
}
```
위 코드에서는 **하나의 트랜잭션 내에서 모든 서비스가 처리되므로** 문제 발생 시 전체가 롤백될 수 있다.

## 3. 트랜잭션을 나눠볼 수 있는 기능 분석
MSA로 전환할 때 트랜잭션을 독립적으로 분리할 수 있는 기능을 찾아야 한다. 아래는 기존 코드에서 트랜잭션을 나눌 수 있는 주요 기능과 설명이다.

### 3.1. 좌석 예약 트랜잭션
```java
public void 좌석예약() {
    콘서트서비스.좌석예약(); // 좌석을 예약하는 서비스 호출
    예약서비스.예약생성(); // 예약 데이터를 생성하는 서비스 호출
}
```
- **설명**: 좌석 예약은 별도의 프로세스로 분리할 수 있으며, 예약 생성은 다른 트랜잭션에서 수행될 수 있다.
- **트랜잭션 분리 가능성**: `콘서트서비스.좌석예약()`과 `예약서비스.예약생성()`을 개별 트랜잭션으로 처리 가능.

### 3.2. 예약 결제 트랜잭션
```java
public void 예약결제() {
    잔액서비스.잔액감소(); // 사용자 잔액 차감
    예약서비스.예약완료(); // 예약 상태를 완료로 변경
    결제서비스.결제처리(); // 결제 시스템 호출
    토큰서비스.만료처리(); // 토큰 만료 처리
}
```
- **설명**: 예약 결제 과정에서 각각의 서비스가 개별적인 DB를 사용할 경우, 트랜잭션을 분리해야 한다.
- **트랜잭션 분리 가능성**:
    1. `잔액서비스.잔액감소()`는 잔액 변경 트랜잭션
    2. `예약서비스.예약완료()`는 예약 상태 변경 트랜잭션
    3. `결제서비스.결제처리()`는 결제 프로세스 트랜잭션
    4. `토큰서비스.만료처리()`는 인증 관련 트랜잭션

각 서비스가 독립적인 트랜잭션을 가지도록 설계할 필요가 있다.

## 4. MSA 전환 후 트랜잭션 처리의 한계
MSA에서는 각 도메인 서비스가 독립적으로 배포되며, 개별적인 데이터 저장소(데이터베이스, 캐시 등)를 가지게 된다. 이로 인해 기존의 단일 트랜잭션 관리 방식이 어려워지며 다음과 같은 문제들이 발생할 수 있다.

### 4.1. 분산 트랜잭션의 부재
MSA에서는 **각 서비스가 독립적인 데이터 저장소를 가지므로** 기존의 단일 트랜잭션을 적용할 수 없다. 예를 들어:
- **잔액서비스**는 `AccountDB`를 사용
- **예약서비스**는 `ReservationDB`를 사용
- **결제서비스**는 `PaymentDB`를 사용

각 데이터베이스가 분리되어 있기 때문에 **하나의 서비스에서 발생한 변경 사항을 다른 서비스와 동기화하는 것이 어려움**.

### 4.2. 데이터 일관성(Consistency) 문제
- 기존 모놀리식에서는 **트랜잭션 롤백이 가능하여 데이터 정합성이 보장**되지만, MSA에서는 개별 서비스 간의 트랜잭션이 분리되어 **동기적인 롤백이 어렵다**.
- 예를 들어, `잔액서비스.잔액감소()`가 성공했지만 `예약서비스.예약완료()`가 실패하면, 잔액이 감소된 상태에서 예약은 처리되지 않는 데이터 불일치 문제가 발생할 수 있다.

### 4.3. 장애 발생 시 롤백 문제
- 모놀리식에서는 하나의 트랜잭션 단위에서 오류가 발생하면 전체 서비스가 자동으로 롤백된다.
- 반면, MSA 환경에서는 **부분적으로 성공한 트랜잭션을 되돌릴 방법이 별도로 필요**하다.
- 예제: 결제가 실패했을 경우, 이미 차감된 잔액을 복원하는 로직을 직접 구현해야 함.

### 4.4. 성능 및 확장성 이슈
- 분산 트랜잭션을 강제하려면 **2PC(Two-Phase Commit)** 같은 방식이 필요하지만, 이는 성능을 저하시킬 수 있다.
- 서비스 간 네트워크 호출이 증가하여 **대기 시간(latency)이 증가하고, 장애 전파 가능성이 높아짐**.

## 5. 결론
MSA 환경에서는 기존 모놀리식에서 제공하던 **단일 트랜잭션 보장**이 어렵고, 서비스 간의 데이터 일관성을 유지하는 것이 주요 도전 과제가 된다.

이를 해결하기 위해서는 새로운 트랜잭션 처리 방식이 필요하며, 이는 후속 문서에서 다루도록 한다.
