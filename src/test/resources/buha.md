# 부하 테스트 선정 및 효율성 고려

## 부하 테스트 대상 선정

| API 엔드포인트 | 선정 이유 |
|--------------|-----------|
| `/api/queue/token` (대기열 토큰 생성) | 대량 요청을 효율적으로 처리할 수 있는지 검증 |
| `/api/concert/{concertId}/schedules` (예약 가능 스케줄 조회) | 다수 사용자의 동시 조회 시 성능 확인 |
| `/api/concert/{concertScheduleId}/seats` (예약 가능 좌석 조회) | 높은 부하 환경에서 정확한 좌석 정보 제공 확인 |
| `/api/balance/{userId}/charge` (잔액 충전) | 분산 락을 활용한 충전 시스템의 안정성 검증 |

---

## 테스트 시나리오 계획

### 1. 대기열 토큰 발급 테스트 (`/api/queue/token`)

**목적**
- 초기 대량 요청 시 대기열 시스템의 처리 성능 검증

**부하 테스트 설정**
- **스케줄링 전략:** `ramping-arrival-rate`
- **VU (가상 사용자):** 50 → 최대 500
- **초기 요청 속도:** 초당 300개 (높은 초기 부하)

**부하 단계**  
| 단계 | 지속 시간 | 초당 요청 수 |
|------|--------|----------|
| 1 | 5초 | 300 |
| 2 | 10초 | 150 |
| 3 | 20초 | 100 |
| 4 | 25초 | 50 |

**성능 임계값**
- **95% 요청 응답 시간**: 750ms 미만
- **요청 실패율**: 최대 5%

---

### 2. 예약 가능 스케줄 및 좌석 조회 테스트

**테스트 준비**
- `setup()`을 통해 사전 API 토큰 발급 후 테스트 진행

**부하 테스트 설정**  
| 단계 | 지속 시간 | 사용자 수 변화 |
|------|--------|--------------|
| 1 | 10초 | 0 → 50 증가 |
| 2 | 10초 | 50 → 100 증가 |
| 3 | 10초 | 100 → 50 감소 |

**성능 임계값**
- **95% 요청 응답 시간**: 500ms 미만
- **요청 실패율**: 1% 미만

## 가상 장애 대응

### 장애 시나리오 1: 대기열 API 응답 지연
**원인**
- 동시 요청 폭주로 인한 서버 리소스 부족
- 대기열 관리 로직 병목 발생

**대응 방안**
- 트래픽 증가 시 로깅 및 알림 시스템 확인
- 추가 서버 자동 확장(Auto Scaling) 적용
- 캐싱 전략 개선 및 대기열 재처리 로직 최적화

### 장애 시나리오 2: 좌석 조회 API 성능 저하
**원인**
- DB 조회 부하 증가
- 캐시 미적용 또는 캐싱 정책 미흡

**대응 방안**
- Redis 캐싱을 활용한 조회 최적화
- DB 쿼리 튜닝 및 인덱스 최적화 적용
- API Gateway의 요청 제한(Rate Limiting) 설정

### 장애 시나리오 3: 잔액 충전 API 충돌 및 데이터 불일치
**원인**
- 분산 락 처리 지연
- 동시 다중 요청으로 인해 충전 금액 불일치 발생 가능

**대응 방안**
- Redisson 기반 분산 락 적용 상태 점검
- 충전 트랜잭션 처리 로직 개선 및 재시도 로직 추가
- 장애 발생 시 알람 및 자동 복구 메커니즘 도입

## 결론 및 개선 사항
✅ **대기열 토큰 발급 API**는 높은 부하에서도 안정적으로 작동  
✅ **예약 가능 스케줄 조회 API**는 빠른 응답 속도 유지  
❌ **좌석 조회 API**의 초기 성능 저하 확인 → 인덱스 최적화 후 개선됨  
✅ **잔액 충전 API**는 분산 락 충돌 없이 안정적으로 처리됨
